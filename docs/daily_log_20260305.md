### 5. OpenMP 对比实验（完成）

为响应导师"与传统编译器优化方法对比"的反馈，实现了 miniMD Force、SPMV、SYMGS 三个 kernel 的 serial + OpenMP 版本，与 LLM 生成的 CUDA kernel 对比。

#### 实验配置
- 编译：`gcc -O2 -g -fopenmp`，8 线程
- Problem size 与 CUDA 实验保持一致

#### 同 Problem Size 对比结果

| Kernel | Problem Size | Serial | OpenMP (8线程) | CUDA (LLM生成) | OpenMP 加速比 | CUDA 加速比 | CUDA/OpenMP |
|--------|-------------|--------|---------------|----------------|-------------|------------|-------------|
| miniMD Force | 100k atoms | 35.2ms | 6.0ms | ~1.5ms | 5.85x | 14.3x | CUDA 快 2.4x |
| SPMV | 100k rows, 27nnz/row | 1.9ms | 0.76ms | ~0.19ms | 2.55x | 10.3x | CUDA 快 4.0x |
| SYMGS | 50k rows, 27 colors | 2.3ms | 0.46ms | ~0.41ms | 4.91x | 5.6x | CUDA 快 1.1x |

#### 分析

1. **miniMD Force**：CUDA 比 OpenMP 快 2.4 倍。embarrassingly parallel 的力计算在 GPU 上受益于大规模并行核心，LLM 生成的 CUDA kernel 有明显附加价值。

2. **SPMV**：CUDA 比 OpenMP 快 4 倍，差距最大。OpenMP 只有 2.55x 加速（8 线程），说明 CPU 上 memory bandwidth 成为瓶颈，GPU 的高带宽（~300 GB/s vs CPU ~50 GB/s）优势在 memory-bound 的 SpMV 上体现最为明显。

3. **SYMGS**：CUDA 和 OpenMP 几乎持平（5.6x vs 4.9x），差距仅 1.1 倍。有数据依赖的代码（Gauss-Seidel）需要多色排序来打破依赖，这限制了 GPU 的并行效率。多色排序在 CPU 多线程上同样有效。

4. **结论**：LLM 生成的 CUDA 代码在 compute-bound（miniMD）和 bandwidth-bound（SPMV）场景下相比 OpenMP 有显著附加价值；在有复杂数据依赖的场景（SYMGS）下优势不明显。这为"何时使用 LLM GPU 代码生成"提供了实践指导。

### 7. Abinit FFT → cuFFT 替换方案分析（方案 B）

由于 Abinit 的实际热点是 FFT 函数（非 nonlop），FFT 有成熟的 GPU 库（cuFFT），因此采用方案 B：不让 LLM 手写 CUDA FFT kernel，而是让 cascaded pipeline 分析代码后建议 cuFFT 替换方案。

#### 源码准备
从 Abinit 源码（`m_sgfft.F90`，5975 行）中提取了 208 行关键代码：
- `sg_fft_cc`：主入口（20 行），展示 3D FFT 的批处理循环
- `sg_fftpx`：x 方向变换签名和数据布局（60 行）
- `sg_ffty`：y 方向变换签名和 OpenMP 模式（60 行）
- `sg_fftrisc`：调度器（58 行），展示 FFT 在 Abinit 中的调用方式

#### 级联分析结果

**Stage 1 (GPT-4o)：** $0.014
- 瓶颈判断：mixed（compute + memory）
- GPU 适合度：True
- 推荐方案：**library_replacement**（LLM 自主判断应使用库替换而非手写 kernel）

**Stage 2 (GPT-5.2)：** $0.065
- 可行性：**high**
- S2 确认 S1 瓶颈判断（未修正）

#### cuFFT 替换方案摘要

核心策略：把 Abinit 分步执行的 1D FFT 合并为 cuFFT 3D batch FFT

| 原函数 | cuFFT 替换 |
|--------|-----------|
| sg_fft_cc（逐 idat 循环） | `cufftPlanMany` + `cufftExecZ2Z` batch 处理 |
| sg_ffty（y 方向 1D） | 不再需要，被 3D cuFFT 覆盖 |
| sg_fftpx（x 方向 + zero-padding） | GPU packing kernel + 3D cuFFT |
| fftrisc（调度器） | GPU scatter → cuFFT → GPU gather |

预期加速比：
- FFT 部分：3-10x（45³网格较小，需 batch 优化）
- 整体：1.8-3.5x（FFT 占 68%，还需考虑传输开销）

实施工时估算：8 步，总计 90-170 小时（生产级集成）

#### 对论文的意义

与 miniMD/SPMV/SYMGS 形成互补的 case study：

| 场景 | LLM 角色 | 结果 |
|------|---------|------|
| miniMD Force（简单并行） | 直接生成 CUDA kernel | 14.3x 加速 |
| SPMV（行并行） | 直接生成 CUDA kernel | 10.3x 加速 |
| SYMGS（数据依赖） | 完整 pipeline 生成 kernel | 5.6x 加速 |
| **Abinit FFT（标准算法）** | **建议 cuFFT 库替换方案** | **方案可行性 high** |

核心论点：LLM 在 GPU 加速中的角色不仅是"写 kernel"，还包括"架构决策"——判断何时该手写自定义 kernel、何时该使用现有 GPU 库。
